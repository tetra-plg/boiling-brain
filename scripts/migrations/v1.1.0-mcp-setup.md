# Migration `v1.1.0-mcp-setup`

Ce slash-command est invoqué automatiquement par `/update-vault` quand le vault est en version < 1.1.0. Il installe la stack MCP + hooks introduite en v1.1 :

- Serveur MCP `boiling-brain-wiki` (FastMCP, 5 outils : `scan_domain`, `preview_page`, `read_page`, `search_wiki`, `drop_to_raw`).
- Hook `Stop` global (`~/.claude/settings.json`) qui détecte commits + fichiers modifiés en fin de session.
- Bloc d'instructions dans `~/.claude/CLAUDE.md` (global) pour que Claude Code utilise le MCP depuis n'importe quel projet.
- Section « Démarrage de session » dans le `CLAUDE.md` du vault, pour qu'au prochain lancement Claude propose `/compress-bb` ou `/ingest` selon les signaux laissés dans `cache/`.

## Contexte

La v1.1 introduit un canal d'accès au vault depuis n'importe quel projet (via MCP user-scope) et une boucle vie-de-session pour ne plus perdre les apprentissages d'une session de travail. Ces deux mécaniques nécessitent des mutations sur des fichiers **globaux user-owned** (`~/.claude/settings.json`, `~/.claude/CLAUDE.md`) — donc une migration **interactive et idempotente** est obligatoire.

Le `CLAUDE.md` du vault (généré au bootstrap depuis `CLAUDE.md.tpl`) n'a pas non plus la section « Démarrage de session » dans les vaults bootstrappés avant v1.1. Sans cette section, Claude ignore les signaux `cache/.session-pending` et `cache/.pending-ingest` même si le hook Stop les écrit correctement.

## Workflow

### 1. Annoncer les nouveautés

Avant toute action, présenter à l'utilisateur ce que la migration va faire :

> **v1.1.0 — MCP + hooks + /compress-bb**
>
> Cette migration installe :
> - Un serveur MCP `boiling-brain-wiki` accessible depuis **n'importe quel projet** (`scan_domain`, `preview_page`, `read_page`, `search_wiki`, `drop_to_raw`).
> - Un hook `Stop` global qui détecte les sessions productives et écrit `cache/.session-pending` dans le vault.
> - Une section « Démarrage de session » dans le `CLAUDE.md` du vault pour que Claude propose `/compress-bb` ou `/ingest` au lancement suivant.
>
> Mutations effectuées :
> - `~/.claude/settings.json` (mcpServers + hooks.Stop)
> - `~/.claude/CLAUDE.md` (bloc `<!-- boiling-brain-wiki-mcp -->`)
> - `<vault>/CLAUDE.md` (section `## Démarrage de session`)
> - `pip install --user fastmcp` si absent
>
> Tout est idempotent et rollbackable.

### 2. AskUserQuestion — décision globale

```json
{
  "questions": [{
    "question": "Activer la stack MCP + hooks v1.1 ?",
    "header": "Setup v1.1",
    "multiSelect": false,
    "options": [

    ]
  }]
}
```

### 3. Branche **« Activer »** — setup-mcp.sh + patch CLAUDE.md vault

#### 3.1. Lancer `setup-mcp.sh`

```bash
bash scripts/setup-mcp.sh --vault-path "$(pwd)"
```

Le script (existant, livré par v1.1) fait :
- Vérifie Python 3 + installe `fastmcp>=2.14,<3` via `pip install --user` si absent.
- Merge l'entrée `mcpServers.boiling-brain-wiki` dans `~/.claude/settings.json` (idempotent).
- Ajoute le hook `Stop` (commande `bash <vault>/scripts/check-session-activity.sh`) dans `~/.claude/settings.json` (idempotent — détecte le path déjà présent).
- Append un bloc avec marqueur `<!-- boiling-brain-wiki-mcp -->` dans `~/.claude/CLAUDE.md` (idempotent — détecte le marqueur).
- `chmod +x` sur `scripts/check-session-activity.sh`.

Capturer le stdout du script et le présenter à l'utilisateur. Si `pip install` échoue (ex: pas de pip, environnement protégé), arrêter et reporter.

#### 3.2. Patcher le `CLAUDE.md` du vault

Lire le `CLAUDE.md` du vault. Si la section `## Démarrage de session` (ou un titre proche : `## Démarrage de session (signaux cache/)`) existe déjà → ne rien faire (idempotence, message « Section déjà présente, skip »).

Sinon, insérer le bloc suivant **avant** la section `## Principes d'écriture` (ou à défaut, à la fin du fichier) :

```markdown
## Démarrage de session (signaux `cache/`)

Au démarrage de chaque session, vérifier les signaux laissés dans `cache/` :

- **`cache/.pending-ingest`** : un ou plusieurs chemins en attente d'ingestion (déposés via le MCP `drop_to_raw` ou par un autre vault). Proposer `/ingest <chemin>` pour chaque entrée. Ne pas supprimer le fichier — attendre la confirmation utilisateur.
- **`cache/.session-pending`** : la session précédente avait des changements non journalisés (commits + fichiers modifiés détectés par le hook `Stop`). Proposer `/compress-bb <slug>` pour archiver le journal dans `raw/notes/sessions/`. Supprimer le fichier après proposition.

Ces vérifications sont silencieuses si les fichiers sont absents.
```

Avant d'écrire, présenter le diff via `AskUserQuestion` (3 options : appliquer / éditer manuellement / skip), comme la migration v1.0.2.

#### 3.3. Commit

Si setup-mcp.sh OK + section ajoutée :

```bash
git add CLAUDE.md
git commit -m "chore: migrate to v1.1.0 (MCP + SessionStart signals)"
```

Si seul le CLAUDE.md a été patché (le setup-mcp.sh ne touche pas à des fichiers du vault, donc rien d'autre à add).

#### 3.4. Confirmer

Afficher :

> ✅ Migration v1.1.0 appliquée.
> - MCP `boiling-brain-wiki` enregistré dans `~/.claude/settings.json`.
> - Hook `Stop` global enregistré.
> - Section « Démarrage de session » ajoutée à `<vault>/CLAUDE.md`.
>
> **Redémarre Claude Code** pour charger le serveur MCP et les hooks.
>
> Teste avec : `/mcp` (doit lister `boiling-brain-wiki`).

### 4. Branche **« Patch CLAUDE.md vault uniquement »**

Identique à 3.2 + 3.3 + 3.4 mais sans 3.1. Le commit message devient :

```
chore: add v1.1 SessionStart signals section to CLAUDE.md
```

### 5. Branche **« Skip »**

Ne rien faire. Indiquer :

> Migration `v1.1.0-mcp-setup` skippée. `.claude/template-version` reste sur la version précédente. La migration sera reproposée au prochain `/update-vault`.

## Idempotence

- **`setup-mcp.sh`** est idempotent par construction : il détecte les entrées déjà présentes (mcpServers, hook Stop, marqueur CLAUDE.md global).
- **Le patch du `CLAUDE.md` du vault** est idempotent via la détection du titre `## Démarrage de session` (insensible à la casse + tolérance sur le suffixe `(signaux cache/)`).
- Si l'utilisateur relance `/update-vault` après une migration partielle, la migration redétecte ce qui est déjà fait et ne propose que les actions restantes.

## Préservation des customizations

- Le `CLAUDE.md` du vault n'est jamais réécrit en entier — seul un bloc nouveau est inséré à un endroit identifié.
- Si l'utilisateur a déjà une section avec un titre proche mais pas exactement `## Démarrage de session`, signaler le conflit potentiel via `AskUserQuestion` plutôt que d'écrire silencieusement.
- `~/.claude/settings.json` et `~/.claude/CLAUDE.md` (globaux) sont mutés via merge ciblé (jamais de réécriture intégrale) — voir `setup-mcp.sh`.

## Rollback

- Hook Stop + entrée mcpServers dans `~/.claude/settings.json` : éditer manuellement le fichier pour retirer les sections `mcpServers.boiling-brain-wiki` et l'entrée correspondante dans `hooks.Stop`.
- Bloc dans `~/.claude/CLAUDE.md` : supprimer le bloc entre les deux marqueurs `<!-- boiling-brain-wiki-mcp -->`.
- Section dans `<vault>/CLAUDE.md` : `git revert` du commit dédié, ou édition manuelle.
- `fastmcp` (pip --user) : `pip uninstall fastmcp` si voulu (inoffensif à laisser).

La procédure complète de rollback en contexte de test pré-release est dans [`docs/local-pre-release-testing.md`](../../docs/local-pre-release-testing.md) Phase 5.
