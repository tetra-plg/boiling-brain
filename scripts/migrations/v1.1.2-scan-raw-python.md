# Migration `v1.1.2` — scan-raw Python rewrite

Invoked automatically by `/update-vault` **step 6**, i.e. **after step 5 has propagated the changed template files**. This migration performs no data transformation on the vault — the wiki, `raw/`, and the index are untouched. Its job is coordination, one behavioral heads-up, and verification.

## Context

`scripts/wiki-maint/scan-raw.sh` used to rebuild its `raw_path → (slug, sha)` index by spawning one `python3` per indexed path plus a `grep|sed` pipeline per frontmatter line, **regardless of scope**. On a mature vault (several hundred `wiki/sources/` pages, several thousand `raw/` files) even a single-file scan exceeded the 120 s timeout, so `/ingest` step 1 (state detection) became unusable and `cache/.pending-ingest` drifted. (#70)

The fix moves all logic into `scripts/wiki-maint/scan-raw.py` (single process, stdlib only). `scan-raw.sh` shrinks to a thin wrapper that keeps the portable `PYTHON_BIN` resolution (#61) and then `exec`s the engine. Default stdout is byte-for-byte identical to the old bash (golden-tested), so **no re-ingest is required** — the vault behaves the same, only fast.

## Files propagated by step 5 (automatic, 3-way merge)

Newly added (case A → `PROPAGATED`, no conflict):

- `scripts/wiki-maint/scan-raw.py` — **the engine** (the wrapper is inert without it).
- `scripts/wiki-maint/scan_raw_fixture.py`, `scripts/wiki-maint/fixtures/scan-raw/default.golden` — parity fixture + frozen golden (tests only).

Modified:

- `scripts/wiki-maint/scan-raw.sh` — 276 → ~50 lines (wrapper).
- `.claude/commands/ingest.md`, `.claude/rules/frontmatter.md`, `scripts/mcp/ingest-headless-guard.sh`, `scripts/mcp/test_wiki_core.py`, `scripts/wiki-maint/test_scan_raw.py`.

## 1. Guarantee the wrapper and engine landed together

**Hard invariant: `scan-raw.sh` (wrapper) is useless without `scan-raw.py` (engine).** If the user deselected either at step 5's file picker, the vault's scan is broken.

Verify both are present and paired on disk:

```bash
test -f scripts/wiki-maint/scan-raw.py \
  && grep -q 'exec .*scan-raw.py' scripts/wiki-maint/scan-raw.sh \
  && echo "OK: wrapper + engine paired" \
  || echo "BROKEN: missing engine or wrapper not pointing at it"
```

If it prints `BROKEN`, re-propagate the missing file from the template before continuing:

```bash
git show "template-upstream/${TARGET_BRANCH}:scripts/wiki-maint/scan-raw.py" > scripts/wiki-maint/scan-raw.py
git show "template-upstream/${TARGET_BRANCH}:scripts/wiki-maint/scan-raw.sh" > scripts/wiki-maint/scan-raw.sh
```

## 2. Conflict on `scan-raw.sh` — prefer the template version

Because the wrapper deletes ~238 lines of now-obsolete bash, step 5 will report `CONFLICT scripts/wiki-maint/scan-raw.sh` for any vault that locally edited that file. **The old scanning logic no longer exists** — local edits to it are meaningless against the wrapper. Recommend **"Use template version (discard local edits)"** for this file unless the vault deliberately forked scan-raw's behavior (rare). A vault-local `$PYTHON_BIN` override still works via the environment; no edit to the wrapper is needed for that.

## 3. Behavioral change to announce (one, intentional)

The engine parses `source_path` / `covered_paths` / `sources:` **only inside the leading `---` frontmatter block**. The old bash scanned the whole file, so those keywords appearing in a page **body** (e.g. a frontmatter example inside a fenced code block) polluted the index and could produce a phantom `SKIP`. Effect on verdicts is **zero unless** such a body keyword happened to name a real `raw/` file — extremely rare. Tell the user this changed; no action needed.

## 4. Post-update health check (run once, report)

Confirm the fix and surface the diagnostics the old bash never computed:

```bash
# a) perf + correctness: should return in well under a second, exit 0
time bash scripts/wiki-maint/scan-raw.sh >/dev/null

# b) index-quality diagnostics (informational — no forced action)
bash scripts/wiki-maint/scan-raw.sh --format=json 2>/dev/null \
  | python3 -c "import sys,json,collections as c; d=json.load(sys.stdin); print('counts:', d['counts']); print('warnings:', dict(c.Counter(w['kind'] for w in d['warnings'])))"

# c) orphans (source pages whose raw disappeared — no auto-deletion)
bash scripts/wiki-maint/scan-raw.sh --orphans 2>/dev/null | grep -c '^ORPHAN' | sed 's/$/ orphan(s)/'
```

Report the numbers to the user. `duplicate-claim`, `missing-sha`, `composite-mismatch`, and orphan counts are **surfaced, not fixed** — a mature vault often shows many (they predate this scan). Cleaning them up is a separate, optional `/lint`-style pass, not part of this migration.

## 5. Optional — reconcile `cache/.pending-ingest`

While scan-raw was timing out, the pending manifest may have accumulated entries that are already ingested. The scan can now identify the purgeable ones (read-only; `purge-pending-ingest.sh` remains the only writer):

```bash
bash scripts/wiki-maint/scan-raw.sh --pending --format=json 2>/dev/null \
  | python3 -c "import sys,json; d=json.load(sys.stdin).get('pending',{}); print('purgeable:', d.get('purgeable',[])); print('stale:', d.get('stale',[]))"
```

Offer to purge those paths via `bash scripts/wiki-maint/purge-pending-ingest.sh <purgeable + stale>`. Skip if the user prefers to handle pending manually.

## Idempotence

Re-running is a no-op: step 5 propagation is content-based (already-current files report `FAST_FORWARD` with no change), the pairing check in §1 is read-only, and the health check in §4 only reads. This migration is **not** `force-rerun` — once applied and recorded in the vault's `applied-migrations`, it does not re-execute.

## Rollback

Restore the pre-migration scan from the vault's own git history:

```bash
git checkout HEAD~1 -- scripts/wiki-maint/scan-raw.sh
git rm scripts/wiki-maint/scan-raw.py scripts/wiki-maint/scan_raw_fixture.py
git rm -r scripts/wiki-maint/fixtures/scan-raw
```

Note this reinstates the #70 timeout on a large vault — rollback only to isolate an unrelated regression.
