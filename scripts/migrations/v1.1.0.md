> `force-rerun: true` — this migration is re-evaluated at every `/update-vault` even when already in `applied-migrations`. Required because Part A's `setup-mcp.sh` invocation has side-effects on user-owned global files (`~/.claude/CLAUDE.md`, `~/.claude/settings.json`) and the v1.1.0 series ships content fixes (e.g. the LLM instructions block was extended from 5 to 12 tools after the #41 MCP tiered-loading refactor — vaults that ran Part A pre-#47 need the script re-run to refresh the block). All parts of this migration are idempotent: re-running is a no-op when the target state is already reached (`setup-mcp.sh` detects existing MCP registration, existing Stop hook entry, existing CLAUDE.md marker with up-to-date content; Part B's stale-flat detection skips silently when the files are already removed).

# Migration `v1.1.0`

This slash-command is invoked automatically by `/update-vault` when the vault is in version < 1.1.0. It performs **two coordinated migrations** that ship together in v1.1.0:

**Part A — MCP + hooks stack** (originally `v1.1.0-mcp-setup`, integrated here):

- MCP server `boiling-brain-wiki` (FastMCP, 5 tools: `scan_domain`, `preview_page`, `read_page`, `search_wiki`, `drop_to_raw`).
- Global `Stop` hook (`~/.claude/settings.json`) that detects commits + modified files at end of session.
- Instructions block in `~/.claude/CLAUDE.md` (global) so Claude Code uses the MCP from any project.
- "Session start" section in the vault's `CLAUDE.md`, so that on the next launch Claude proposes `/compress-bb` or `/ingest` based on the signals left in `cache/`.

## Context

v1.1 introduces a channel to access the vault from any project (via user-scope MCP) and a session lifecycle loop to stop losing the learnings of a working session. Both mechanics require mutations on **user-owned global files** (`~/.claude/settings.json`, `~/.claude/CLAUDE.md`) — so an **interactive and idempotent** migration is mandatory.

The vault's `CLAUDE.md` (generated at bootstrap from `CLAUDE.md.tpl`) does not have the "Session start" section either in vaults bootstrapped before v1.1. Without that section, Claude ignores the signals `cache/.session-pending` and `cache/.pending-ingest` even if the Stop hook writes them correctly.

## Workflow

### 1. Announce what's new

Before any action, present to the user what the migration will do:

> **v1.1.0 — MCP + hooks + /compress-bb**
>
> This migration installs:
> - An MCP server `boiling-brain-wiki` accessible from **any project** (`scan_domain`, `preview_page`, `read_page`, `search_wiki`, `drop_to_raw`).
> - A global `Stop` hook that detects productive sessions and writes `cache/.session-pending` in the vault.
> - A "Session start" section in the vault's `CLAUDE.md` so that Claude proposes `/compress-bb` or `/ingest` on the next launch.
>
> Mutations performed:
> - MCP server registered via `claude mcp add -s user` (user-scope, not stored in `settings.json`).
> - `~/.claude/settings.json` (`hooks.Stop` only).
> - `~/.claude/CLAUDE.md` (`<!-- boiling-brain-wiki-mcp -->` block).
> - `<vault>/CLAUDE.md` (`## Session start` section).
> - `pipx install fastmcp` if absent (fallback `pip install --user`).
>
> Everything is idempotent and rollbackable.

### 2. AskUserQuestion — global decision

```json
{
  "questions": [{
    "question": "Enable the v1.1 MCP + hooks stack?",
    "header": "v1.1 setup",
    "multiSelect": false,
    "options": [
      {"label": "✅ Enable", "description": "Runs setup-mcp.sh (mutates settings.json + global CLAUDE.md) and patches the vault CLAUDE.md. Idempotent."},
      {"label": "✏️ Patch vault CLAUDE.md only", "description": "Skip the MCP server and the global Stop hook. Just add the Session start section to the vault's CLAUDE.md. Useful if you already installed MCP another way or don't want any global mutation."},
      {"label": "⏭️ Skip for this session", "description": "Touch nothing. The migration will be re-proposed at the next /update-vault. .claude/template-version stays at 1.0.x."}
    ]
  }]
}
```

### 3. Branch **"Enable"** — setup-mcp.sh + vault CLAUDE.md patch

#### 3.1. Run `setup-mcp.sh`

```bash
bash scripts/mcp/setup-mcp.sh --vault-path "$(pwd)"
```

The script (existing, shipped by v1.1) does:
- Checks Python 3 and the `claude` command (Claude Code CLI).
- Installs `fastmcp` if absent — pipx is prioritized (clean, isolated venv), pip --user as fallback. If both fail (PEP 668 + no pipx), explicit error.
- Resolves the Python interpreter that can actually `import fastmcp`: when pipx is used, it's the python from the pipx venv (`$PIPX_LOCAL_VENVS/fastmcp/bin/python`), **not** the system `python3` — this is the fix for the import-fastmcp error on PEP 668 systems.
- Registers the MCP server via `claude mcp add -s user boiling-brain-wiki -e WIKI_PATH=<vault> -- <pipx-venv-python> <vault>/scripts/mcp/mcp-wiki.py` (idempotent — detects via `claude mcp get`). User-scope means the server is visible from every project. The server now exposes **12 tools** (5 originals + 7 `scan_<type>` granular drill-downs added in v1.1.0 final).
- Adds the `Stop` hook (command `bash <vault>/scripts/hooks/check-session-activity.sh`) to `~/.claude/settings.json` (idempotent — detects the path already present).
- Writes a block with marker `<!-- boiling-brain-wiki-mcp -->` to `~/.claude/CLAUDE.md`, describing all 12 MCP tools and the tiered loading pattern (`scan_domain` → `scan_<type>(query)` → `preview_page` → `read_page`). **Self-healing**: if the marker is already present but the block is the pre-#47 version (5 tools only), the script detects and replaces it in place. No-op if the block is already current.
- `chmod +x` on `scripts/hooks/check-session-activity.sh`.

Capture the script's stdout and present it to the user. If the fastmcp install or `claude mcp add` fails, stop and report.

> **Note for vaults already migrated to v1.1.0 (pre-#47)**: if your vault has `v1.1.0` in `applied-migrations` of `.claude/template-version` but was migrated before the MCP tiered-loading refactor (#47), `/update-vault` will not re-trigger this migration (slug-deduplicated). Two options to refresh the global `~/.claude/CLAUDE.md` block:
> - **Simple**: re-run `bash <vault>/scripts/mcp/setup-mcp.sh --vault-path "$(pwd)"` directly. The script's self-healing logic detects and replaces the outdated 5-tool block with the new 12-tool + tiered-loading version.
> - **Full re-run**: edit `.claude/template-version` to remove the `v1.1.0` line from `applied-migrations`, then re-run `/update-vault`. The full Part A workflow will execute again (idempotent on its other steps).
> The MCP tools (scan_concepts, scan_entities, etc.) themselves are auto-discovered by Claude Code at session start — only the LLM-facing instructions (the global block) need this refresh.

#### 3.2. Patch the vault's `CLAUDE.md`

Read the vault's `CLAUDE.md`. If the `## Session start` section (or a close title: `## Session start (cache/ signals)`, or any localized variant such as `## Démarrage de session`) already exists → do nothing (idempotence, message "Section already present, skip").

Otherwise, insert the following block **before** the `## Writing principles` section (or, failing that, at the end of the file). Render the block in the vault language declared in `CLAUDE.md` — the snippet below is the EN reference; translate to the vault language if different.

```markdown
## Session start (`cache/` signals)

At the start of each session, check the signals left in `cache/`:

- **`cache/.pending-ingest`**: paths awaiting ingestion. Run `bash scripts/wiki-maint/scan-raw.sh` first — purge silently the `SKIP` (stale, already ingested), suggest `/ingest <path>` for `NEW` / `MODIFIED` entries. Remove the file if empty.
- **`cache/.session-pending`**: the previous session had unjournaled changes (commits + modified files detected by the `Stop` hook). Propose `/compress-bb <slug>` to archive the journal into `raw/notes/sessions/`. Delete the file after proposal.

These checks are silent if the files are absent.
```

Before writing, present the diff via `AskUserQuestion` (3 options: apply / edit manually / skip), like the v1.0.2 migration.

#### 3.3. Commit

If setup-mcp.sh OK + section added:

```bash
git add CLAUDE.md
git commit -m "chore: migrate to v1.1.0 (MCP + SessionStart signals)"
```

Only `CLAUDE.md` is added (setup-mcp.sh does not touch any vault file).

#### 3.4. Confirm

Display:

> ✅ Migration v1.1.0 applied.
> - MCP `boiling-brain-wiki` registered (user scope, via `claude mcp`).
> - Global `Stop` hook registered in `~/.claude/settings.json`.
> - "Session start" section added to `<vault>/CLAUDE.md`.
>
> **Restart Claude Code** to load the MCP server and hooks.
>
> Test with: `/mcp` (should list `boiling-brain-wiki`).

### 4. Branch **"Patch vault CLAUDE.md only"**

Same as 3.2 + 3.3 + 3.4 but without 3.1. The commit message becomes:

```
chore: add v1.1 SessionStart signals section to CLAUDE.md
```

### 5. Branch **"Skip"**

Do nothing. State:

> Migration `v1.1.0` skipped. `.claude/template-version` stays on the previous version. The migration will be re-proposed at the next `/update-vault`.

## Idempotence

- **`setup-mcp.sh`** is idempotent by construction: it detects entries already present (MCP server via `claude mcp get`, Stop hook command, global CLAUDE.md marker).
- **The vault `CLAUDE.md` patch** is idempotent via detection of the `## Session start` heading (case-insensitive + tolerance for the `(cache/ signals)` suffix and localized variants like `## Démarrage de session`).
- If the user re-runs `/update-vault` after a partial migration, the migration re-detects what is already done and only proposes the remaining actions.

## Preserving customizations

- The vault's `CLAUDE.md` is never rewritten as a whole — only a new block is inserted at an identified location.
- If the user already has a section with a close-but-not-exact title, signal the potential conflict via `AskUserQuestion` rather than writing silently.
- `~/.claude/settings.json` and `~/.claude/CLAUDE.md` (globals) are mutated via targeted merge (never wholesale rewrite) — see `setup-mcp.sh`.

## Rollback

- **MCP server**: the entry is no longer in `settings.json` (since it's registered via `claude mcp add`). Remove it with `claude mcp remove boiling-brain-wiki -s user`.
- **Stop hook in `~/.claude/settings.json`**: edit the file manually to remove the entry under `hooks.Stop` whose command points to `<vault>/scripts/hooks/check-session-activity.sh`.
- **Block in `~/.claude/CLAUDE.md`**: delete the block between the two `<!-- boiling-brain-wiki-mcp -->` markers.
- **Section in `<vault>/CLAUDE.md`**: `git revert` the dedicated commit, or edit manually.
- **`fastmcp`** (pipx or pip --user): `pipx uninstall fastmcp` (or `pip uninstall fastmcp`) if desired (harmless to leave).

## Part B — Scripts layout reorganisation (new in v1.1.0)

The template's `scripts/` directory has moved to a feature-based layout. Existing vaults still have the old flat layout (`scripts/extract-frames.sh`, `scripts/scan-raw.sh`, etc.). Without this migration, `/update-vault` would propagate the new files under `scripts/<feature>/` and leave the old flat files in place, producing duplicates.

### Workflow

1. **Detect the stale flat paths**. For each of the following paths in the vault, if it exists as a flat file at the root of `scripts/`, mark it for removal:

   - `scripts/extract-frames.sh`
   - `scripts/sample-frames.sh`
   - `scripts/diff-frames.py`
   - `scripts/transcribe.sh`
   - `scripts/backfill-summaries.py`
   - `scripts/enrich-hub.py`
   - `scripts/scan-raw.sh`
   - `scripts/scan-domain-refs.sh`
   - `scripts/mcp-wiki.py`
   - `scripts/setup-mcp.sh`
   - `scripts/check-session-activity.sh`

   `scripts/sync-repos.sh` stays at the root (unchanged). `scripts/migrations/` stays at the root (unchanged).

2. **Show the user the list of stale files** that will be removed, and ask for confirmation:

   ```
   The following stale flat-layout script files will be `git rm`'d from the vault:
   <list>
   They have been replaced upstream by the new feature-based layout in `scripts/<feature>/`,
   which step 5 of /update-vault will propagate automatically.
   Proceed? [y/N]
   ```

3. **If confirmed, `git rm` the stale files** in the vault. If declined, stop the migration here without marking it as applied (the user can re-run `/update-vault` later).

4. **Re-align global Claude Code config to the new script paths**. Three places hold absolute paths that point at the old flat layout and must be re-aligned:

   - The **MCP server registration** in `claude mcp` registry (was `<vault>/scripts/mcp-wiki.py`, now `<vault>/scripts/mcp/mcp-wiki.py`).
   - The **Stop hook entry** in `~/.claude/settings.json` (was `bash <vault>/scripts/check-session-activity.sh`, now `bash <vault>/scripts/hooks/check-session-activity.sh`). Without cleanup, `setup-mcp.sh` adds a new entry at the new path but leaves the old one in place — Claude Code would then fire the stale entry too and fail because Step 3 already `git rm`'d the old script.
   - The **block in `~/.claude/CLAUDE.md` global** (under the `<!-- boiling-brain-wiki-mcp -->` marker): no path is hard-coded inside the block, so it does NOT need rewriting. `setup-mcp.sh` will detect the marker and skip it. Mentioned here only for clarity.

   **Step 4.a — Clean the stale Stop hook entry in `~/.claude/settings.json`.**

   Read `~/.claude/settings.json`. Under `hooks.Stop`, locate and remove any entry whose `command` (or any nested `hooks[].command`) contains the substring `scripts/check-session-activity.sh` WITHOUT the `hooks/` segment in front (i.e. `<vault>/scripts/check-session-activity.sh` but NOT `<vault>/scripts/hooks/check-session-activity.sh`). Preserve unrelated Stop hooks. If `~/.claude/settings.json` doesn't exist or has no such stale entry, skip silently.

   Concretely (the agent executing this migration should do this in-place via its edit tools):
   - Open `~/.claude/settings.json`.
   - Iterate `settings["hooks"]["Stop"]`; drop any item whose direct or nested command matches the substring `/scripts/check-session-activity.sh` AND does NOT match `/scripts/hooks/check-session-activity.sh`.
   - Write the result back (JSON, 2-space indent, `ensure_ascii=False` if Python is used).

   **Step 4.b — Re-register the MCP server.**

   Display these commands to the user (do not run them automatically — they affect global config):

   ```bash
   claude mcp remove boiling-brain-wiki -s user || true
   bash <ABSOLUTE_VAULT_PATH>/scripts/mcp/setup-mcp.sh --vault-path <ABSOLUTE_VAULT_PATH>
   ```

   Substitute `<ABSOLUTE_VAULT_PATH>` with the actual absolute path of the user's vault (e.g. `/Users/<user>/Workspace/<vault-name>`).

   `setup-mcp.sh` is idempotent and reuses Part A's full setup path: it resolves the right Python interpreter (pipx venv first to avoid the `Failed to reconnect: -32000` crash on PEP 668 systems, fallback to system `python3` only if `import fastmcp` works there), re-registers the MCP server via `claude mcp add -s user` with the correct `-e WIKI_PATH=<vault>`, appends the new Stop hook entry (since 4.a removed the stale one, the new entry will be the only one), and leaves the `~/.claude/CLAUDE.md` global block untouched (marker-detected).

5. **Confirm the user has re-aligned the global config** before marking the migration as applied. The user can verify with:

   - `/mcp` inside Claude Code (after a Claude Code restart) — `boiling-brain-wiki` should appear without a `Failed to reconnect` error.
   - `grep -c "scripts/check-session-activity.sh" ~/.claude/settings.json` should return `0` if you don't have any other unrelated vault registered, OR each remaining match should include `/scripts/hooks/check-session-activity.sh` (the new path).

   If the user has not re-aligned (or skips), the migration is **not** marked as applied — the user will be re-prompted at the next `/update-vault`.

### Idempotence notes

- If a stale flat file does not exist (already removed in a previous attempt), skip it silently.
- Re-running step 4 is safe: step 4.a is idempotent (no-op if no stale entry left), `claude mcp remove … || true` swallows the "not found" case, and `setup-mcp.sh` is idempotent across all its operations (MCP register, Stop hook append, CLAUDE.md global block append).
- Part A (MCP + hooks install) and Part B (scripts reorg) are independent: re-running either is safe.

## Part C — Template-owned ADRs moved from `wiki/decisions/` to `docs/` (v1.1.0 finalize)

The 3 ADRs that previously lived in `wiki/decisions/` of the template (`extraction-frames-induction-runbook.md`, `tracked-repos-immutable-snapshots.md`, `ingest-video-modes-a-b-generalisation.md`) have been moved to `docs/` upstream. They are architectural decisions of the **template project itself**, not of the user's vault — `wiki/` is now reserved for the user's own ADRs (each vault enriches `wiki/decisions/` with decisions specific to its own domains).

Vaults migrated to v1.1.0 before this move still have these 3 files at the old location (`wiki/decisions/`). Without this cleanup, they would coexist with the new `docs/` copies propagated by step 5 — duplication.

### Workflow

1. **Detect the stale paths** in the vault:

   - `wiki/decisions/extraction-frames-induction-runbook.md`
   - `wiki/decisions/tracked-repos-immutable-snapshots.md`
   - `wiki/decisions/ingest-video-modes-a-b-generalisation.md`

   For each, if the file exists at the vault path, mark it for removal. If none exists (already removed in a previous run, or vault bootstrapped post-move), skip Part C silently.

2. **Show the user the list** and ask for confirmation:

   ```
   These 3 template-owned ADRs were moved from wiki/decisions/ to docs/ upstream.
   The wiki/decisions/ versions in your vault are now duplicates of the docs/ versions
   propagated by step 5. Remove them from wiki/decisions/?

       wiki/decisions/extraction-frames-induction-runbook.md
       wiki/decisions/tracked-repos-immutable-snapshots.md
       wiki/decisions/ingest-video-modes-a-b-generalisation.md

   (Your own ADRs in wiki/decisions/, if any, are untouched.) Proceed? [y/N]
   ```

3. **If confirmed, `git rm` the listed files** (skipping any that don't exist). If declined, leave them in place — the user can handle the duplication manually.

   ```bash
   for f in \
     wiki/decisions/extraction-frames-induction-runbook.md \
     wiki/decisions/tracked-repos-immutable-snapshots.md \
     wiki/decisions/ingest-video-modes-a-b-generalisation.md; do
     [ -f "$f" ] && git rm "$f"
   done
   ```

4. **Dedicated commit** for the cleanup:

   ```bash
   git commit -m "chore: remove template-owned ADRs from wiki/decisions/ (moved to docs/ upstream)"
   ```

### Idempotence notes

- If the 3 files are already absent from `wiki/decisions/` (new vault, or Part C re-run), the step is a no-op.
- The user's own `wiki/decisions/*.md` (their own ADRs) are never touched — only the 3 template-owned files are removed by name.
- `wiki/decisions/decision.md.tpl` (the ADR template for the user) remains in `wiki/decisions/` — unaffected.
