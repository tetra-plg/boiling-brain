# Migration `v1.1.2`

Invoked automatically by `/update-vault` **step 6** (after **step 5** has propagated the changed template files), for any vault in version < 1.1.2. This single migration covers both v1.1.2 changes:

- **Part A — scan-raw Python rewrite (#70)**: coordination + health check for the already-propagated `scan-raw.py` / `scan-raw.sh`. It performs no data transformation on the vault — the wiki, `raw/`, and the index are untouched. Read-only checks plus one optional purge — **never blocks the verdict**.
- **Part B — Domain orientation (#71)**: inserts the `## Domain orientation` section into existing `<domain>-expert.md` agents and adds the `wiki-cli.sh` allowlist rule to the vault's `.claude/settings.json`. Interactive (every change goes through `AskUserQuestion`) — **drives the verdict**.

Run **Part A first** (it verifies what step 5 just propagated), then **Part B**.

## Part A — scan-raw Python rewrite (#70)

### A0. Context

`scripts/wiki-maint/scan-raw.sh` used to rebuild its `raw_path → (slug, sha)` index by spawning one `python3` per indexed path plus a `grep|sed` pipeline per frontmatter line, **regardless of scope**. On a mature vault (several hundred `wiki/sources/` pages, several thousand `raw/` files) even a single-file scan exceeded the 120 s timeout, so `/ingest` step 1 (state detection) became unusable and `cache/.pending-ingest` drifted. (#70)

The fix moves all logic into `scripts/wiki-maint/scan-raw.py` (single process, stdlib only). `scan-raw.sh` shrinks to a thin wrapper that keeps the portable `PYTHON_BIN` resolution (#61) and then `exec`s the engine. Default stdout is byte-for-byte identical to the old bash (golden-tested), so **no re-ingest is required** — the vault behaves the same, only fast.

### A1. Files propagated by step 5 (automatic, 3-way merge)

Newly added (case A → `PROPAGATED`, no conflict):

- `scripts/wiki-maint/scan-raw.py` — **the engine** (the wrapper is inert without it).
- `scripts/wiki-maint/scan_raw_fixture.py`, `scripts/wiki-maint/fixtures/scan-raw/default.golden` — parity fixture + frozen golden (tests only).

Modified:

- `scripts/wiki-maint/scan-raw.sh` — 276 → ~50 lines (wrapper).
- `.claude/commands/ingest.md`, `.claude/rules/frontmatter.md`, `scripts/mcp/ingest-headless-guard.sh`, `scripts/mcp/test_wiki_core.py`, `scripts/wiki-maint/test_scan_raw.py`.

### A2. Guarantee the wrapper and engine landed together

**Hard invariant: `scan-raw.sh` (wrapper) is useless without `scan-raw.py` (engine).** If the user deselected either at step 5's file picker, the vault's scan is broken.

Verify both are present and paired on disk:

```bash
test -f scripts/wiki-maint/scan-raw.py \
  && grep -q 'exec .*scan-raw.py' scripts/wiki-maint/scan-raw.sh \
  && echo "OK: wrapper + engine paired" \
  || echo "BROKEN: missing engine or wrapper not pointing at it"
```

If it prints `BROKEN`, re-propagate the missing file from the template before continuing:

```bash
git show "template-upstream/${TARGET_BRANCH}:scripts/wiki-maint/scan-raw.py" > scripts/wiki-maint/scan-raw.py
git show "template-upstream/${TARGET_BRANCH}:scripts/wiki-maint/scan-raw.sh" > scripts/wiki-maint/scan-raw.sh
```

### A3. Conflict on `scan-raw.sh` — prefer the template version

Because the wrapper deletes ~238 lines of now-obsolete bash, step 5 will report `CONFLICT scripts/wiki-maint/scan-raw.sh` for any vault that locally edited that file. **The old scanning logic no longer exists** — local edits to it are meaningless against the wrapper. Recommend **"Use template version (discard local edits)"** for this file unless the vault deliberately forked scan-raw's behavior (rare). A vault-local `$PYTHON_BIN` override still works via the environment; no edit to the wrapper is needed for that.

### A4. Behavioral change to announce (one, intentional)

The engine parses `source_path` / `covered_paths` / `sources:` **only inside the leading `---` frontmatter block**. The old bash scanned the whole file, so those keywords appearing in a page **body** (e.g. a frontmatter example inside a fenced code block) polluted the index and could produce a phantom `SKIP`. Effect on verdicts is **zero unless** such a body keyword happened to name a real `raw/` file — extremely rare. Tell the user this changed; no action needed.

### A5. Post-update health check (run once, report)

Confirm the fix and surface the diagnostics the old bash never computed:

```bash
# a) perf + correctness: should return in well under a second, exit 0
time bash scripts/wiki-maint/scan-raw.sh >/dev/null

# b) index-quality diagnostics (informational — no forced action)
bash scripts/wiki-maint/scan-raw.sh --format=json 2>/dev/null \
  | python3 -c "import sys,json,collections as c; d=json.load(sys.stdin); print('counts:', d['counts']); print('warnings:', dict(c.Counter(w['kind'] for w in d['warnings'])))"

# c) orphans (source pages whose raw disappeared — no auto-deletion)
bash scripts/wiki-maint/scan-raw.sh --orphans 2>/dev/null | grep -c '^ORPHAN' | sed 's/$/ orphan(s)/'
```

Report the numbers to the user. `duplicate-claim`, `missing-sha`, `composite-mismatch`, and orphan counts are **surfaced, not fixed** — a mature vault often shows many (they predate this scan). Cleaning them up is a separate, optional `/lint`-style pass, not part of this migration.

### A6. Optional — reconcile `cache/.pending-ingest`

While scan-raw was timing out, the pending manifest may have accumulated entries that are already ingested. The scan can now identify the purgeable ones (read-only; `purge-pending-ingest.sh` remains the only writer):

```bash
bash scripts/wiki-maint/scan-raw.sh --pending --format=json 2>/dev/null \
  | python3 -c "import sys,json; d=json.load(sys.stdin).get('pending',{}); print('purgeable:', d.get('purgeable',[])); print('stale:', d.get('stale',[]))"
```

Offer to purge those paths via `bash scripts/wiki-maint/purge-pending-ingest.sh <purgeable + stale>`. Skip if the user prefers to handle pending manually.

### A7. Rollback (Part A only)

Restore the pre-migration scan from the vault's own git history:

```bash
git checkout HEAD~1 -- scripts/wiki-maint/scan-raw.sh
git rm scripts/wiki-maint/scan-raw.py scripts/wiki-maint/scan_raw_fixture.py
git rm -r scripts/wiki-maint/fixtures/scan-raw
```

Note this reinstates the #70 timeout on a large vault — rollback only to isolate an unrelated regression.

## Part B — Domain orientation (#71)

### B0. Context

v1.1.2 gives each domain-expert subagent first-class, self-service knowledge of its domain through the already-shipped tiered-loading CLI (`scripts/mcp/wiki-cli.py`, wrapped by the new portable `scripts/mcp/wiki-cli.sh`). Instead of depending on a flat page-title list injected by the main context at every spawn, the agent runs `scan-domain → scan-<type> → preview/read` itself, with a `Glob`/`Grep` fallback.

For new vaults, `domain-expert.md.tpl` already carries the section, so bootstrap handles it. For **existing** vaults, the per-domain `<domain>-expert.md` files are **user-owned** (customized at bootstrap, potentially enriched by `/evolve-agent` afterwards). This migration inserts the section **additively** — it never regenerates or reorders an agent's prompt — and every change goes through `AskUserQuestion`.

Two things ship with Part B:

1. The `## Domain orientation` section, inserted into each detected domain-expert agent (with `{{domain_slug}}` resolved to the agent's real slug).
2. A `Bash(bash scripts/mcp/wiki-cli.sh *)` rule in `.claude/settings.json` `permissions.allow`, so the experts' orientation queries run without a first-use permission prompt in interactive sessions. (The `.claude/settings.json` of a vault is a local accumulation of approvals — it is not propagated by the template, so this migration is its only delivery channel.)

**Vault language.** Existing per-domain agents are fully localized to the vault language (an FR vault's agent has `## Ta mission`, `## Ce que tu produis`, … not the EN template headings). Like `v1.0.3-vault-language.md`, this migration **renders the section prose in the agent's own language** — only the `bash scripts/mcp/wiki-cli.sh …` commands, paths, flags, and `[[wikilink]]` examples stay verbatim. Placement anchors are detected by role, not by an exact English heading, so the migration works whatever the vault language.

### B1. Add the `wiki-cli.sh` allowlist rule to `settings.json`

Read `.claude/settings.json`. If it does not exist, treat the current allow list as empty. The rule to ensure present, verbatim:

```
Bash(bash scripts/mcp/wiki-cli.sh *)
```

**Idempotence**: if `permissions.allow` already contains this exact string, do nothing (report "already present"). Otherwise append it to `permissions.allow`, preserving every existing entry and the rest of the file (`env`, `additionalDirectories`, other keys) untouched. If the file is absent, create it with the minimal shape:

```json
{
  "permissions": {
    "allow": ["Bash(bash scripts/mcp/wiki-cli.sh *)"]
  }
}
```

Ask via `AskUserQuestion`:

### B4. Insert the section into each agent

For each detected domain-expert agent, with its captured `<slug>`:

**Idempotence check first.** If the file already contains a line `## Domain orientation` → skip it (already migrated, or manually added). Report it as "already present".

Otherwise, insert the section below. Two substitutions apply:

- **`<slug>` → the agent's real slug** everywhere it appears (the template's `{{domain_slug}}` is already a concrete slug in an existing agent file).
- **Prose → the agent's language.** Detect the agent's language from its existing headings/body (e.g. `## Ta mission` ⇒ French). Translate every prose sentence and bullet into that language. Keep **verbatim**, untranslated: every `bash scripts/mcp/wiki-cli.sh …` command, every path, every flag (`--query`, `--top`, `--json`), the `[[wikilink]]` token, `scan-domain`/`scan-<type>`/`preview`/`read`/`search` subcommand names, and the `## Ingest summary` reference. If the agent is in English, use the text as-is. Keep the heading itself as `## Domain orientation` (a stable, language-independent anchor the idempotence check relies on).

Canonical (English) source text:

```markdown
## Domain orientation

Your first reflex, before reading the source: orient yourself in your own domain. You have the wiki's tiered-loading CLI — use it instead of guessing.

- If your spawn prompt includes a `scan-domain` snapshot, use it as your starting map.
- Otherwise run it yourself: `bash scripts/mcp/wiki-cli.sh scan-domain <slug>` (hub, counts by type, top pages by centrality — ~1k tokens).
- Drill down only where the source demands it: `bash scripts/mcp/wiki-cli.sh scan-concepts <slug> --query "<topic>"` (same for `scan-entities`, `scan-cheatsheets`, `scan-syntheses`, `scan-decisions`, `scan-diagrams`, `scan-sources`).
- `bash scripts/mcp/wiki-cli.sh preview <page>` before deciding create-vs-enrich; `read <page>` only when you actually enrich it.
- **Never guess whether a page exists** — check with a scan `--query` first. This applies to every `[[wikilink]]` you write: verify the target exists, otherwise create it (if the threshold is met) or log a Radar item.
- Cross-domain anchors (co-ingest, cross-refs): `bash scripts/mcp/wiki-cli.sh search "<term>"`.
- Idempotence step 0: resolve the candidate source page with `bash scripts/mcp/wiki-cli.sh scan-sources <slug> --query "<slug-of-source>"` then `preview` it (frontmatter → `source_sha256`), instead of guessing its path.
- If `wiki-cli` fails (missing interpreter, non-zero exit, or a denied command in headless mode), fall back to `Glob`/`Grep` and mention the degradation in your `## Ingest summary`.
- These are all the commands you need; for occasional flags (`--top`, `--json`), run `bash scripts/mcp/wiki-cli.sh <command> --help`.
```

**Placement (language-agnostic).** Insert it as a new top-level section immediately **before the deliverables section** — the section that enumerates the wiki pages the agent produces (`## What you produce` in the template, or its localized equivalent such as `## Ce que tu produis` / `## Lo que produces` / `## Was du produzierst`; identify it by role — the heading whose body lists `wiki/sources`, `wiki/concepts`, `wiki/entities` deliverables — not by an exact English string). If you cannot confidently identify that section (heavily restructured agent), fall back to inserting immediately **after** the frontmatter block (the closing `---`) and note the non-standard placement in the report. Never split an existing section.

**Optional mission-bullet rewrite.** If the agent's mission section (`## Your mission` / localized equivalent such as `## Ta mission`) still describes the old title-list context — a bullet listing the existing pages of the domain, in whatever language — propose replacing that bullet with this one, rendered in the agent's language (commands/paths verbatim):

```markdown
- The **current wiki context**: a fresh `scan-domain` snapshot of your domain (hub, counts by type, top pages by centrality) and the contents of `wiki/domains/<slug>.md`. If the snapshot is absent, produce it yourself (see **Domain orientation** below).
```

(Same slug substitution.) If the mission bullet was customized away or you cannot confidently locate the old wording, leave it as-is — the Domain orientation section is self-contained.

**Review one by one**: for each file, show the insertion diff (the new section + its anchor context) and ask Apply / Edit manually / Skip.

**Apply to all**: insert into every detected agent that lacks the section, log the count.

### B5. Preserving user customizations

`.claude/agents/<domain>-expert.md` may contain user customizations (extra sections, refined rules, `/evolve-agent` output). This migration only **inserts one new section** (and optionally rewrites the single stale mission bullet), rendered in the agent's own language. It never deletes, reorders, or rewrites any other content. If the deliverables section cannot be confidently identified (in any language), it falls back to a safe post-frontmatter insertion and reports it, rather than guessing a location inside existing prose.

## Idempotence (both parts)

- **Part A** (read-only): re-running is a no-op. Step 5 propagation is content-based (already-current files report `FAST_FORWARD` with no change), the pairing check (A2) is read-only, the health check (A5) only reads, and the pending oracle (A6) is read-only (`purge-pending-ingest.sh` remains the only writer).
- **Part B — settings.json**: skip if `permissions.allow` already contains `Bash(bash scripts/mcp/wiki-cli.sh *)` (exact-string match).
- **Part B — agents**: skip any agent whose file already contains a `## Domain orientation` heading. A partial migration can therefore be safely re-run by the next `/update-vault` without double-inserting.

This migration is **not** `force-rerun` — once applied and recorded in the vault's `applied-migrations`, it does not re-execute.

## Commit

If **Part B** modified at least one file:

```bash
git add .claude/agents/*.md .claude/settings.json wiki/log.md

```

If Part B modified nothing (everything already present or skipped), do not commit. Part A never commits on its own (its optional `.pending-ingest` purge lives under `cache/`, which is not tracked).

## Verdict for `/update-vault`

Part A is **never blocking** (read-only checks + one optional purge). The verdict is driven by **Part B**:

- If **all** Part B changes were Applied (settings.json rule present + every detected agent has the section) → migration is **complete** → `/update-vault` may bump `.claude/template-version` to 1.1.2.
- If **any** Part B change was skipped or edited manually → migration is **partial** → `/update-vault` must NOT bump `.claude/template-version`. The migration is re-proposed at the next `/update-vault` and skips what is already present (see _Idempotence_).
